# ADR 001 - OpenClaw hardening

## Status

Accepted.

## Context

The OpenClaw integration had overlapping discovery endpoints, local bridge routes with mutable operations, duplicated model-selection logic, and a global catalog cache without an entry cap.

## Decisions

- Keep Bearer authentication mandatory for mutable local bridge routes (`/api/model`, `/api/config/model`, `/v1/chat/completions`). The token is generated by the CLI, stored in `openclaw.json`, sent in the WS `hello`, and reused by HTTP fallback routes.
- Keep a single shared bridge token instead of inventing a second HTTP-only secret. The bridge is loopback-bound and origin-gated, and duplicating secrets would add rotation and recovery failure modes without materially improving local security.
- Keep explicit model metadata tables in `server/lib/openclaw-model-metadata.ts` ahead of string heuristics. Heuristics are allowed only as conservative fallback.
- Use the shared `selectPrimaryModelRef` algorithm for CLI sync/runtime flows so `--model`, current config, server presets, and catalog order resolve consistently.
- Prefer `/openclaw/manifest` for new clients. Legacy `/openclaw/discovery`, `/openclaw/catalog`, `/openclaw/status`, and `/openclaw/config` remain for one release with deprecation headers.
- Bound the OpenClaw catalog cache with a manual LRU cap and absolute TTL to avoid unbounded growth in persistent runtimes.

## Consequences

- Older CLIs keep working during the compatibility window.
- New CLIs can make one manifest request instead of composing several overlapping endpoints.
- Corrupt provider credentials are surfaced as degraded state instead of being only logged.
- Any future gateway protocol simplification must be coordinated with OpenClaw upstream before removing `gateway-auth.mjs`.
